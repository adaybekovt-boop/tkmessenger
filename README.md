# orbits_flutter

Flutter port of `git_push/` (React + Vite + PeerJS). Started 2026-04-20.

## Status of the port

Legend: ✅ real implementation · 🟡 skeleton + port notes · ⬜ not started

- ✅ `base64_helpers.dart` — mirrors `src/core/base64.js`
- ✅ `crypto_local.dart` — vault KDF + AES-GCM, mirrors `src/core/crypto.js` (PBKDF2 SHA-256, 600k iters v3)

- ⬜ `wire_session.dart` — session registry, handshake, teardown
- ⬜ `identity_key.dart`, `prekey_bundle.dart`, `vault_kek.dart`, `scrypt_kdf.dart`

### Peer (`lib/peer/`)
- ✅ `signaling.dart` — host list, backoff, ICE config, mirrors `src/peer/signaling.js`
- 🟡 `peer_connection_manager.dart` — skeleton matches `src/peer/peerConnectionManager.js` (441 lines)
- ⬜ `peerjs_client.dart` — **must be written from scratch**. No official PeerJS client for Dart. Options below.
- ⬜ `packet_router.dart`, `multi_tab_lock.dart`

### PeerJS — three possible paths

1. **Port the PeerJS wire protocol to Dart.** Most faithful. ~300-500 LoC WebSocket client speaking the PeerJS JSON signaling protocol + wrapping `flutter_webrtc` data channels. Lets us reuse existing PeerJS server pool (`0.peerjs.com`, …) unchanged.
2. **Replace PeerJS with a custom signaling server + raw WebRTC.** Cleaner long-term but means hosting our own signaling. Breaks interop with the React web build unless both sides migrate.
3. **Use `flutter_peerjs` (third-party, ~300 stars).** Fast to start but the package is thinly maintained. Acceptable for a prototype, risky for prod.

Recommended: **option 1**, done after the crypto layer is solid and tested.

## CI / GitHub release builds

The `.github/workflows/build.yml` workflow produces a ready-to-install
**Android APK** and **Windows EXE bundle** on every push. Platform folders
(`android/`, `windows/`) aren't checked in — CI generates them via
`flutter create` and patches in our permissions + signing config before
building.

### How to use it

1. Push the project to a GitHub repo. The workflow runs automatically on
   every push to `main`/`master` and on every pull request. Build
   artifacts are attached to the workflow run page (Actions → run →
   Artifacts).

2. To cut a versioned release:
   ```
   git tag v0.1.0
   git push origin v0.1.0
   ```
   This triggers the `release` job, which collects the artifacts from
   the Android + Windows jobs and attaches them to a GitHub Release.
   Direct download links appear on the repo's *Releases* page.

3. Manually trigger a build any time from Actions → "Build & Release" →
   "Run workflow".

### Android signing — the workaround

The release APK is signed with the **Android debug keystore** instead of
a real upload key. Reasons:

- No Play Store key set up (and we don't ship through Play anyway).
- The debug keystore is auto-generated by the JDK and cached across CI
  runs (via `actions/cache`), so the signature is **stable** — installing
  v0.2.0 over v0.1.0 works without uninstalling first.
- APKs install fine via `adb install`, "Open with package installer" on
  the device, or by tapping the file in a file manager (you may need to
  toggle "Install from unknown sources" once for whichever app you used
  to open it — that's normal for any sideloaded APK).

If you ever do want a real signing key:
1. Run `keytool -genkey -keystore upload.jks ...` locally.
2. Base64-encode it and add it to the repo as a GitHub Secret called
   `ANDROID_KEYSTORE_B64`, plus secrets for `KEY_ALIAS`, `KEY_PASSWORD`,
   `STORE_PASSWORD`.
3. Edit `tool/ci/patch_android.sh` to consume those env vars and write
   them into a `key.properties` file.

### Windows EXE — avoiding "VCRUNTIME140.dll missing"

Flutter's Windows runner links against the MSVC C++ runtime
(`VCRUNTIME140.dll`, `MSVCP140.dll`). Machines without VC++
Redistributable installed throw the missing-DLL popup at startup. The
workflow copies those DLLs out of the GitHub-hosted runner's VS
install and bundles them inside the zip alongside the .exe — so the
zip is self-contained, just unzip and run.

### Outputs

| Artifact              | What's inside                                                            |
|-----------------------|--------------------------------------------------------------------------|
| `orbits-android.zip`  | `orbits-android-arm64.apk` (most modern phones), `-arm32`, `-x86_64`, `-universal` |
| `orbits-windows.zip`  | `orbits_flutter.exe` + Flutter engine DLL + VC++ runtime DLLs + `data/`  |
